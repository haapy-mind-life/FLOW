1. 로컬 저장소에 수정 사항 반영하기

먼저, 코드 문서와 UX 문서에서 언급된 수정 사항을 실제 코드에 반영해야 합니다.

1-1. 수정 사항 확인하기

  • 코드 문서와 UX 문서를 읽고 어떤 부분을 수정해야 하는지 파악합니다.
  • 싱잉볼 모드 추가
  • 종료 알람의 페이드 아웃 기능 제거
  • 배경 음악의 페이드 아웃 및 페이드 인 시간 조정 (10초)
  • 연속 모드 기능 개선
  • 디자인 및 UX 수정 사항 적용

1-2. 코드 수정하기

  • HTML 파일: 새로운 버튼 추가 및 속성 수정
  • CSS 파일: 새로운 스타일 추가 및 기존 스타일 수정
  • JavaScript 파일: 새로운 기능 구현 및 기존 기능 수정

1-3. 코드 테스트하기

  • 브라우저에서 수정된 웹사이트를 열어 모든 기능이 정상적으로 동작하는지 확인합니다.
  • 싱잉볼 모드 정상 작동 여부
  • 배경 음악과 알람의 페이드 인/아웃 올바른지 확인
  • 디자인과 사용자 인터페이스 기대대로 표시되는지 확인

2. Git 저장소에 수정 사항 커밋하기

수정된 코드를 Git 저장소에 반영합니다.

2-1. 변경 사항 확인하기

터미널(또는 명령 프롬프트)을 열고 프로젝트 폴더로 이동한 후, 다음 명령어를 입력하여 변경 사항을 확인합니다.

git status

수정된 파일 목록이 표시됩니다.

2-2. 변경 사항 스테이징하기

수정된 파일들을 Git에 추가합니다.

git add .

. 은 현재 디렉토리의 모든 변경된 파일을 추가하겠다는 의미입니다.

2-3. 변경 사항 커밋하기

변경 사항을 커밋합니다.

git commit -m "싱잉볼 모드 추가 및 배경 음악 페이드 타임 조정 (2023-10-03 수정사항 반영)"

커밋 메시지는 변경 사항을 명확히 설명할 수 있도록 작성합니다.

3. 원격 저장소에 변경 사항 푸시하기

로컬 저장소의 변경 사항을 GitHub 원격 저장소에 반영합니다.

git push origin main

origin은 원격 저장소의 기본 이름이며, main은 브랜치 이름입니다.
만약 브랜치 이름이 master라면 git push origin master를 사용하세요.

4. GitHub에서 코드 확인하기

웹 브라우저에서 GitHub 리포지토리(https://github.com/haapy-mind-life/FLOW)로 이동하여 변경 사항이 반영되었는지 확인합니다.
수정된 파일들이 업데이트되어 있어야 합니다.

5. Git 릴리즈 생성하기

새로운 버전의 릴리즈를 생성하여 사용자들에게 변경 사항을 알립니다.

5-1. 버전 태그 생성하기

git tag -a v1.1.0 -m "Release version 1.1.0 - 싱잉볼 모드 추가 및 기능 개선"

버전 번호는 프로젝트의 버전 관리 정책에 따라 지정합니다.
일반적으로 MAJOR.MINOR.PATCH 형식을 사용합니다.

  • MAJOR: 큰 변화나 호환성에 영향이 있는 경우 증가
  • MINOR: 기능 추가나 개선이 있는 경우 증가
  • PATCH: 버그 수정이 있는 경우 증가

여기서는 기능 추가이므로 MINOR 버전을 증가시켜 v1.1.0으로 설정했습니다.

5-2. 태그를 원격 저장소에 푸시하기

git push origin v1.1.0

생성한 태그를 원격 저장소에 푸시하여 GitHub에서 인식할 수 있도록 합니다.

5-3. GitHub에서 릴리즈 생성하기

  1. GitHub 리포지토리 페이지에서 상단 메뉴의 "Releases" 탭을 클릭합니다.
  2. "Draft a new release" 버튼을 클릭합니다.
  3. "Tag version"에서 방금 생성한 태그(v1.1.0)를 선택합니다.
  4. "Release title"에 릴리즈 제목을 입력합니다.
    예: Release v1.1.0 - 싱잉볼 모드 추가 및 기능 개선
  5. "Describe this release" 섹션에 릴리즈 노트를 작성합니다.
    코드 문서와 UX 문서의 수정 내역을 참고하여 주요 변경 사항을 작성합니다.
### 변경 사항
- 싱잉볼 모드 추가: 배경 음악 없이 1분 타이머와 종료 알람 제공
- 배경 음악 페이드 인/아웃 시간 조정 (10초)
- 종료 알람의 페이드 아웃 기능 제거
- UX 개선 및 디자인 수정
  1. 필요한 경우 바이너리 파일이나 추가 자료를 "Attach binaries" 섹션에 업로드합니다.
  2. "Publish release" 버튼을 클릭하여 릴리즈를 완료합니다.

6. 릴리즈 확인 및 공유하기

GitHub 리포지토리의 "Releases" 페이지에서 새로운 릴리즈가 생성되었는지 확인합니다.
릴리즈 페이지의 URL을 복사하여 사용자들과 공유할 수 있습니다.
예: https://github.com/haapy-mind-life/FLOW/releases/tag/v1.1.0

7. GitHub Pages에 변경 사항 반영하기 (웹사이트 배포)

만약 GitHub Pages를 통해 웹사이트를 호스팅하고 있다면, 변경 사항이 자동으로 반영됩니다.

7-1. GitHub Pages 설정 확인하기

  • 리포지토리의 "Settings" 탭으로 이동합니다.
  • 왼쪽 메뉴에서 "Pages"를 선택합니다.
  • "Branch"가 main (또는 master)으로 설정되어 있고, "/ (root)" 폴더를 선택했는지 확인합니다.

7-2. 웹사이트에서 변경 사항 확인하기

브라우저에서 GitHub Pages URL로 이동하여 변경 사항이 적용되었는지 확인합니다.
예: https://haapy-mind-life.github.io/FLOW/
만약 변경 사항이 반영되지 않았다면, 몇 분 정도 기다린 후 페이지를 새로고침해보세요.

8. 앞으로의 Git 사용 팁

  • 자주 커밋하고 푸시하기: 작업한 내용은 작은 단위로 자주 커밋하고 원격 저장소에 푸시하세요.
  • 커밋 메시지 작성: 변경 사항을 명확히 알 수 있도록 커밋 메시지를 작성하세요.
  • 태그 활용하기: 중요한 버전마다 태그를 생성하여 버전 관리를 효율적으로 할 수 있습니다.
  • 릴리즈 노트 작성하기: 릴리즈마다 변경 사항을 기록하여 사용자들과 공유하세요.

9. 요약

  • 코드 수정: 코드 문서와 UX 문서의 수정 사항을 실제 코드에 반영하고 테스트합니다.
  • Git에 반영: 변경된 파일들을 Git에 추가하고 커밋합니다.
  • 원격 저장소에 푸시: 로컬 변경 사항을 GitHub 원격 저장소에 푸시합니다.
  • 릴리즈 생성: 새로운 버전 태그를 만들고 GitHub에서 릴리즈를 생성하여 사용자들과 공유합니다.
  • GitHub Pages 업데이트: 웹사이트가 최신 상태로 업데이트되었는지 확인합니다.
